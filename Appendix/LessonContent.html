GDPR Basics

Why does GDPR exist?

GDPR exists to protect people's personal data, in a world where software systems constantly collect and process information about their users.

Before GDPR, companies were able to:

  • Collect extensive/unnecessary user data,
  • Store it indefinitely,
  • Share it without transparency.

GDPR changes this, by enforcing transparency and responsibility.

How does this affect developers?

As a developer, you are often the one to:

  • Design databases,
  • Define APIs,
  • Decide what user activity to log,
  • Integrate third party systems

This means that you are directly influencing if a system is compliant or not! Because of that, it is important you understand exactly what GDPR regulations entails.

What is personal data?

Personal data is any information which could identify a person, directly (e.g. name, phone number, email address) or indirectly (e.g. IP address, location data).

Practical Example

{
    "username": "john123",
    "ip": "192.168.1.1",
    "lastLogin": "2026-03-30"
}

While john123 is not a real name, the ip address combined with the user activity could identify an individual, making it personal data.

What is special data?

Some data is more sensitive, making it more strictly regulated. GDPR Article 9 lists the following special categories:

  • Racial or ethnic origin
  • Political opinions
  • Religious or philosophical beliefs
  • Trade union membership
  • Genetic data
  • Biometric data
  • Health data
  • Sexual orientation

Processing this kind of data is prohibited by default unless an Article 9 exception applies. If your system handles it, you need:

  • Explicit consent or another valid exception
  • Higher security measures
  • A Data Protection Impact Assessment (DPIA)

Quiz: GDPR Awareness

Q1: What is the main goal of the GDPR?

  • To protect people's personal data
    Correct. The GDPR was created to protect people's personal data and give them control over how it is collected and processed.
  • To prevent companies from collecting any data
    Incorrect. GDPR does not ban data collection. It regulates how personal data is collected and processed.
  • To make all user data publicly available
    Incorrect. GDPR aims to protect personal data, not make it public.

Q2: Why should developers care about GDPR?

  • Developers decide what data to collect and how to store it
    Correct. Developers design databases, define APIs and decide what to log. These choices directly affect whether a system is GDPR compliant.
  • GDPR only applies to managers and legal teams
    Incorrect. GDPR affects everyone involved in building systems that handle personal data. Developers play a key role.
  • Developers are never held responsible for data handling
    Incorrect. Developers directly influence compliance through the systems they build. Poor design choices can lead to violations.

Quiz: Personal and Special Data

Q1: Which of the following is considered personal data under GDPR?

  • A user's IP address
    Correct. GDPR Article 4(1) defines personal data as any information that can identify a person directly or indirectly. An IP address is an online identifier that can be used to identify someone.
  • The name of a company
    Incorrect. A company name is not personal data. GDPR protects natural persons, not legal entities.
  • A public weather forecast
    Incorrect. A weather forecast does not relate to an identifiable person.

Q2: A JSON object stores a username and an IP address. Is this personal data?

  • Yes, the combination can identify a person
    Correct. Even if a username is not a real name, the IP address combined with user activity can identify an individual. This makes the whole record personal data.
  • No, because a username is not a real name
    Incorrect. Personal data includes any information that could identify someone directly or indirectly. An IP address alone can qualify.
  • Only if the user gives consent
    Incorrect. Whether data counts as personal data does not depend on consent. It depends on whether the data can identify a person.

Q3: Which of the following is a special category of data under GDPR Article 9?

  • Home address
    Incorrect. A home address is personal data but not a special category. Special categories are listed in Article 9 and include health data, genetic data, biometric data, racial/ethnic origin, political opinions, religious beliefs, trade union membership and sexual orientation.
  • Health data
    Correct. Health data is explicitly listed as a special category in Article 9 of the GDPR. Processing it is prohibited unless a specific exception applies.
  • Email address
    Incorrect. An email address is personal data but not a special category.
  • Phone number
    Incorrect. A phone number is personal data but not a special category.

Q4: What extra requirement applies when processing special category data?

  • You need explicit consent or another Article 9 exception
    Correct. Processing special category data is prohibited by default. You need both a lawful basis under Article 6 and a separate condition under Article 9 (such as explicit consent) to process it.
  • You only need to encrypt it
    Incorrect. Encryption is a good security measure but it is not sufficient on its own. You need a valid legal basis and an Article 9 exception.
  • No extra requirements apply
    Incorrect. Special category data has stricter rules. Processing it without meeting Article 9 conditions is a violation.

Data Protection

GDPR Data Protection Principles

When working with personal data, GDPR defines a set of core principles that must guide your system design:

  • Lawfulness - You must have a valid reason to process user data
  • Data Minimisation - Only collect what you actually need
  • Integrity & Confidentiality - Keep data secure
  • Accuracy - Ensure data is correct and up to date
  • Accountability - You are responsible for everything you do with data

These principals directly affect how you design your systems, databases, APIs and much more!

Lawfulness

You cannot collect or process data without a valid legal reason. This is called a lawful basis.

Examples include:

  • User consent
  • Fulfilling a contract
  • Legal obligations

For the developer, this means that each and every API endpoint or data field should have a clear reason to exist.

Data Minimisation

You should only collect the data you actually need, and nothing more.

Less data means:

  • Lower risk of data breaches
  • Easier compliance
  • Better performance and storage efficiency

Practical Example

You should not store a user's full date of birth if you only need to verify they are over 18.

You should instead store a simple flag, such as isAdult = true.

Integrity & Confidentiality (Security)

You must protect personal data from:

  • Unauthorized access
  • Data leaks
  • Loss or corruption

Examples of good practices:

  • Encrypt sensitive data
  • Use HTTPS
  • Hash passwords
  • Limit access (roles/permissions)

If a system gets breached, developers are usually the ones being held responsible, so save yourself some trouble and learn how to protect personal data!

Accuracy

Personal data must be correct and up-to-date.

When interacting with your system, users should be able to update their data or fix any incorrect information.

Practical example

If a user changes their email, but your system still sends emails to the old one, then you are violating this principle.

Accountability

On top of having to follow GDPR regulation, you must also be able to prove that you do.

This means:

  • Logging data processing activities
  • Documenting decisions
  • Having clear policies

If someone audits your system, you should be able to explain every piece of data you store. If anything goes wrong or is not up to regulation, you could be held responsible!

Data Minimisation

Data minimisation means you should only collect and store the data that is strictly necessary for a system/feature to function.

Practical Example

Bad design:

{
  "name": "John Doe",
  "email": "john@email.com",
  "phone": "+12345678",
  "dateOfBirth": "1999-01-01",
  "address": "Street 123"
}

For a simple account, most of this data is unnecessary.

Better design:

{
  "email": "john@email.com"
}

As a rule of thumb, you should start minimal, and expand only when needed. The less data you have, the smaller the risk in case of a breach! As an added benefit, you also have a simpler system to manage!

Purpose Limitation

You must clearly define why you collect data. This also means that you cannot reuse it for something else later.

Practical Example

  • User gives email for account → OK
  • Using that email for marketing → NOT OK (without consent)

Storage Limitation

You cannot store personal data forever. Old data should be either deleted (if no longer required) or archived (if required by law)

Practical Example:

  • Delete inactive accounts after X time
  • Automatically clean logs

Quiz: Core Principles

Q1: Which GDPR principle states that you must have a valid legal reason to process personal data?

  • Data minimisation
    Incorrect. Data minimisation means only collecting what you need. Having a valid legal reason falls under the lawfulness principle.
  • Lawfulness
    Correct. Under Article 5(1)(a), personal data must be processed lawfully. This means you need a valid legal reason (lawful basis) before processing.
  • Accountability
    Incorrect. Accountability means being able to prove you comply with GDPR. Lawfulness is about having a valid legal reason.
  • Storage limitation
    Incorrect. Storage limitation is about not keeping data longer than needed.

Q2: What does the accountability principle require?

  • You must follow GDPR and be able to prove it
    Correct. Article 5(2) states that the controller must be responsible for and be able to demonstrate compliance with all the other principles.
  • You must delete all data after 30 days
    Incorrect. There is no fixed 30-day rule in GDPR. The storage limitation principle says data should not be kept longer than necessary, but the period depends on the purpose.
  • You must ask users for consent before every action
    Incorrect. Consent is one of six lawful bases. Accountability is about proving you follow the rules, not about obtaining consent for every action.

Q3: A system stores passwords in plain text. Which principle does this violate?

  • Data minimisation
    Incorrect. Data minimisation is about collecting only what you need. Storing passwords in plain text is a security issue.
  • Integrity and confidentiality
    Correct. Article 5(1)(f) requires that data is processed securely. Storing passwords in plain text fails to protect data from unauthorized access.
  • Purpose limitation
    Incorrect. Purpose limitation means data should only be used for the purpose it was collected for.
  • Accuracy
    Incorrect. Accuracy means keeping data correct and up-to-date. This is about security, not correctness.

Q4: Under GDPR Article 5, how many core data protection principles are there?

  • 5
    Incorrect. There are 7 principles: lawfulness/fairness/transparency, purpose limitation, data minimisation, accuracy, storage limitation, integrity/confidentiality, and accountability.
  • 6
    Incorrect. There are 7 principles defined in Article 5.
  • 7
    Correct. Article 5 defines 7 principles: lawfulness/fairness/transparency, purpose limitation, data minimisation, accuracy, storage limitation, integrity/confidentiality, and accountability.
  • 8
    Incorrect. There are 7 principles in Article 5. You might be thinking of the 8 data subject rights.

Quiz: Data Handling

Q1: A registration form asks for name, email, phone, home address and date of birth. The service only needs an email to function. Which principle is violated?

  • Accuracy
    Incorrect. Accuracy is about keeping data correct. The issue here is that too much data is being collected.
  • Data minimisation
    Correct. Article 5(1)(c) states that data must be limited to what is necessary. If only an email is needed, collecting phone, address and date of birth violates this principle.
  • Accountability
    Incorrect. Accountability is about proving compliance. The primary violation here is collecting unnecessary data.
  • Lawfulness
    Incorrect. Lawfulness is about having a legal reason to process data. The issue here is the amount of data collected.

Q2: A user signs up with their email for an account. The company later uses that email to send marketing newsletters without asking. Which principle is violated?

  • Storage limitation
    Incorrect. Storage limitation is about how long data is kept. The issue here is that data is being used for a different purpose.
  • Integrity and confidentiality
    Incorrect. The data was not leaked or accessed without authorization. It was used for a different purpose.
  • Purpose limitation
    Correct. Article 5(1)(b) states that data must be collected for specified purposes and not used for anything incompatible. Using an account email for marketing without consent is a new purpose.
  • Accuracy
    Incorrect. The email address is accurate. The problem is that it is being used for an unrelated purpose.

Q3: How long can you store personal data according to GDPR?

  • Forever, as long as the user agreed once
    Incorrect. Consent does not grant indefinite storage. The storage limitation principle still applies.
  • Only as long as necessary for the stated purpose
    Correct. Article 5(1)(e) states that data should be kept only as long as necessary for the purpose it was collected for. After that, it must be deleted or anonymized.
  • A maximum of 5 years
    Incorrect. GDPR does not define a fixed time limit. The retention period depends on the purpose and must be justified.
  • Until the user requests deletion
    Incorrect. While users have a right to erasure, you should not wait for a request. Data must be deleted when it is no longer needed.

Lawful Principles

Lawful Bases

Before you store or process any personal data, you must answer one question: "Why am I allowed to do this?". This justification is called a lawful basis.

If you cannot clearly answer this, then you should probably not be collecting the data in the first place!

There are 6 Lawful Bases covered by the GDPR (which we will cover in the following lessons), but usually only one is needed to justify data collection!

Legal Obligation

You can process personal data if it is necessary to comply with a legal obligation.

This means the law requires you to perform a task, and processing personal data is part of fulfilling that requirement.

Practical Examples

Example 1:

You receive an official request from law enforcement to provide specific personal data.

  → You must comply and share the required data.

Example 2:

You send employee salary information to the tax authority.

  → This is required for tax law compliance.

Important:

You can only use this when a specific law applies, not as a general excuse.

Public Interest

You can process personal data if it is necessary for a task carried out in the public interest. This usually applies to government or official institutions, not standard applications.

Practical Examples

Example 1:

A government system processes citizen data for public services.

  → This is required to provide those services.

Example 2:

Health authorities track disease spread.

  → This helps protect public health.

Vital Interest

You can process personal data if it is necessary to protect someone's life. This only applies in emergency situations, rarely used in normal applications.

Practical Examples

Example 1:

Medical staff access patient data during an emergency when the patient is unconscious.

  → The data is needed to provide treatment.

Example 2:

Emergency services use stored health information.

  → This can prevent serious harm.

Consent

You can process personal data if the user has given clear permission. This is used when the feature is optional and the user has a real choice.

Practical Examples

Example 1:

A user signs up for a newsletter.

  → They agree to receive emails.

Example 2:

A user accepts tracking cookies for analytics.

  → Data is collected based on their permission.

Important:

Users must be able to withdraw consent at any time!

Contractual Necessity

You can process personal data if it is necessary to perform a contract with the user. This means the data is required to deliver a service the user has requested.

Practical Examples

Example 1:

You store a user's email and password to allow login.

  → Without this data, the account cannot function.

Example 2:

You collect a shipping address for an online order.

  → The product cannot be delivered without it.

Legitimate Interest

You can process personal data if you have a valid reason, and it does not outweigh the user's rights.

This requires a simple 3-step check.

Step 1: Do you have a legitimate interest?

  → It must be real, specific, and your own (e.g. security, fraud prevention).

Step 2: Is the processing necessary?

  → Can you achieve the same goal with less data?

Step 3: Do your interests outweigh the user's rights?

  → Would the user reasonably expect this?

Important

If any of these steps fail, you cannot use this legal basis.

Practical Examples

Example 1:

You log user activity to detect fraud or abuse.

  → This helps protect your system.

Example 2:

You monitor system usage to improve performance.

  → This helps keep the application stable.

This basis is flexible, but also the easiest to misuse.

Getting Consent

When you rely on consent, you must ask for it in a clear and honest way. Users should understand exactly what they are agreeing to.

Be clear:

  → Say what data you collect and why.

Be specific:

  → Separate different purposes (e.g. marketing vs. analytics).

Give a real choice:

  → No pre-checked boxes or forced consent.

Allow withdrawal:

  → Users must be able to change their mind.

Practical Examples

Example 1:

A user signs up for a newsletter using an unchecked checkbox.

  → They actively choose to receive emails.

Example 2:

A cookie banner asks permission before tracking users.

  → Tracking only starts after acceptance.

Important

If consent is not freely given, it is not valid!

Privacy Policy

A privacy policy explains how and why you process personal data. It must be easy to find and easy to understand.

Practical Examples

Example 1:

A website provides a clear privacy page linked in the footer.

  → Users can access it at any time.

Example 2:

An app explains what data is collected during signup.

  → Users are informed before using the service.

What to Include

  • Data collected: What personal data you store.
  • Purpose: Why you collect the data.
  • Legal basis: Why you are allowed to process it.
  • User rights: How users can access or delete their data.
  • Contact info: How users can reach you.

Important

If users don't understand your policy, it is not good enough.

Clarity is more important than legal complexity.

Quiz: Bases

Q1: How many lawful bases does GDPR Article 6 define for processing personal data?

  • 4
    Incorrect. Article 6 defines 6 lawful bases: consent, contract, legal obligation, vital interest, public interest and legitimate interest.
  • 6
    Correct. Article 6 lists 6 lawful bases: consent, contract, legal obligation, vital interest, public interest and legitimate interest.
  • 8
    Incorrect. There are 6 lawful bases. You may be thinking of the 8 data subject rights.
  • 10
    Incorrect. Article 6 defines exactly 6 lawful bases.

Q2: A hospital accesses a patient's medical records during an emergency when the patient is unconscious. Which lawful basis applies?

  • Consent
    Incorrect. The patient is unconscious and cannot give consent. A different basis must be used.
  • Vital interest
    Correct. Article 6(1)(d) allows processing when it is necessary to protect someone's life. This applies in emergency situations where the person cannot give consent.
  • Legitimate interest
    Incorrect. While a hospital has an interest in treating patients, the vital interest basis is the correct one for life-threatening emergencies.
  • Public interest
    Incorrect. Public interest applies to tasks carried out by public authorities. Treating an individual patient in an emergency falls under vital interest.

Q3: An online store collects a shipping address to deliver an order. Which lawful basis applies?

  • Consent
    Incorrect. The address is required to fulfill the order. Since the user placed the order, this falls under contractual necessity rather than consent.
  • Contractual necessity
    Correct. Article 6(1)(b) allows processing that is necessary to perform a contract. The user requested a delivery, so collecting the shipping address is required to fulfill that contract.
  • Legal obligation
    Incorrect. No law requires collecting a shipping address. The basis here is the contract between the store and the customer.
  • Legitimate interest
    Incorrect. Since there is a direct contractual relationship, contractual necessity is the more appropriate basis.

Q4: Is consent always required to process personal data?

  • Yes, consent is always mandatory
    Incorrect. Consent is one of six lawful bases. If another basis applies (like a contract or legal obligation), consent may not be needed.
  • No, there are 5 other lawful bases that can justify processing
    Correct. Consent is one option. GDPR also allows processing based on contract, legal obligation, vital interest, public interest or legitimate interest.
  • Only for special category data
    Incorrect. Special category data does require an Article 9 condition (which may include explicit consent), but regular personal data also needs a lawful basis. That basis does not have to be consent.

Quiz: Consent

Q1: Which of the following is NOT valid consent under GDPR?

  • An unchecked checkbox the user actively selects
    Incorrect. An unchecked checkbox that the user actively selects is valid. It represents a clear, affirmative action.
  • A pre-checked checkbox that the user does not uncheck
    Correct. Recital 32 of the GDPR states that silence or pre-ticked boxes do not constitute consent. The user must take a clear affirmative action.
  • A user typing 'I agree' in a form field
    Incorrect. Typing 'I agree' is a clear affirmative action and can count as valid consent.

Q2: What must a privacy policy include?

  • Only the company name
    Incorrect. GDPR Articles 13 and 14 require much more information in a privacy policy.
  • What data is collected, why, the legal basis, user rights and contact info
    Correct. Articles 13 and 14 require that you inform users about the data you collect, the purpose, the lawful basis, their rights and how to contact you.
  • A list of all employees who access user data
    Incorrect. You do not need to list individual employees. You do need to explain who the data controller is and provide contact details.
  • The company's annual revenue
    Incorrect. Revenue is not required in a privacy policy.

Q3: A cookie banner starts tracking users before they click 'Accept'. Is this GDPR compliant?

  • Yes, because the banner is visible
    Incorrect. Showing a banner is not enough. Tracking before consent is given violates the consent requirement.
  • No, tracking can only start after the user gives consent
    Correct. Under GDPR, non-essential cookies require user consent before activation. Tracking before the user clicks 'Accept' means consent was never obtained.
  • It depends on the type of cookies
    Partially true but misleading. Strictly necessary cookies (like session cookies) do not need consent. However, analytics and tracking cookies always require consent first.

Privacy & Data Rights

User Rights Overview

Under GDPR, users keep control over their personal data. This means your system must support a set of user rights.

The 8 Rights

  • Right to be informed
  • Right of access
  • Right to rectification
  • Right to restriction
  • Right to object
  • Right to erasure
  • Right to data portability
  • Rights related to automated decision-making

Important

These are not optional features. Your system must support them, where applicable.

Right to be Informed

Users have the right to know what data you collect and why. This is usually done through a privacy policy.

Practical Examples

Example 1:

You explain data collection during sign-up, so users know what they agree to.

Example 2:

You provide a clear 'privacy policy' page. This way, users can read it anytime.

Right of Access

Users can request a copy of the personal data you store about them.

Practical Examples

Example 1:

A user downloads their account data.

  → Data is provided in a readable format.

Example 2:

A user asks what data is stored.

  → You must provide the information.

Developer Implications

You must be able to locate and export user data from your system, so keep that in mind when design your system!

Right to Rectification

Users can correct inaccurate/incomplete data.

Practical Examples

Example 1:

A user updates their email address.

  → Stored data should be corrected.

Example 2:

A user reports incorrect personal details.

  → You should fix the data.

Important

Incorrect data is a compliance risk, so be careful to keep it accurate and up-to-date!

Right to Restriction

Users can ask you to limit how their data is used, instead of deleting it. This is often used when users want to keep their data, but stop certain processing.

When It Applies

  • The user claims the data is incorrect
  • The processing is unlawful, but the user does not want deletion
  • You no longer need the data, but the user does
  • You are verifying a deletion or objection request

How to Implement

  • Disable access to the data
  • Stop using it in features or processing
  • Temporarily hide or isolate the data

Practical Examples

Example 1:

A user disputes incorrect data.

  → Processing is paused until it is verified.

Example 2:

A user needs data for a legal claim.

  → You keep the data, but stop using it.

Right to Object

Users can object to certain types of data processing. This is common for situations such as marketing or tracking.

Practical Examples

Example 1:

A user opts out of marketing emails.

  → You must stop sending them.

Example 2:

A user objects to analytics tracking.

  → Tracking must stop.

Right to Erasure

Users can request that their personal data is deleted, also known as the right to be forgotten.

Practical Examples

Example 1:

A user deletes their account.

  → Their data is removed from your system.

Example 2:

A user withdraws consent.

  → Related data must be deleted.

Important

Deletion should be complete and not reversible.

Right to Data Portability

Users can receive their data in a format that can be reused elsewhere, such as JSON or CSV.

When It Applies

  • The data was provided by the user
  • The processing is based on consent or a contract
  • The data is processed automatically (i.e. not paper-based)

How to Implement

  • Provide export formats (e.g. JSON, CSV)
  • Ensure data is structured and reusable
  • Allow users to download or transfer their data

Practical Examples

Example 1:

A user downloads their data as JSON.

  → It can be imported into another system.

Example 2:

A user requests their data to be sent to another provider.

  → You transfer the data directly.

Automated Decision-Making

Users have the right not to be subject to fully automated decisions that significantly affect them.

This includes systems that use algorithms or AI to make important decisions.

When It Is Allowed

  • It is required by law
  • It is necessary for a contract
  • The user has given explicit consent

What You Must Provide

  • Clear information about how decisions are made
  • A way to request human review
  • A way to challenge the decision

Practical Examples

Example 1:

An algorithm approves or rejects a loan application.

  → This decision impacts the user directly.

Example 2:

A system profiles users to determine pricing or access.

  → Users may be treated differently based on automation.

Quiz: User Rights

Q1: How many data subject rights does the GDPR define?

  • 5
    Incorrect. The GDPR defines 8 data subject rights, covered in Articles 12 through 22.
  • 6
    Incorrect. There are 8 rights. You might be thinking of the 6 lawful bases from Article 6.
  • 8
    Correct. The GDPR grants 8 rights: to be informed, access, rectification, erasure, restriction, portability, to object and rights related to automated decision-making.
  • 10
    Incorrect. There are exactly 8 data subject rights.

Q2: A user asks for a copy of all personal data you store about them. Which right are they exercising?

  • Right to erasure
    Incorrect. Erasure (Article 17) is about deleting data. Requesting a copy falls under a different right.
  • Right of access
    Correct. Article 15 gives users the right to obtain a copy of their personal data and information about how it is being processed.
  • Right to data portability
    Incorrect. Portability (Article 20) is about receiving data in a reusable format to transfer to another service. A general request for a copy of stored data is the right of access.
  • Right to rectification
    Incorrect. Rectification (Article 16) is about correcting inaccurate data.

Q3: A user notices their email address is wrong in your system and asks you to fix it. Which right applies?

  • Right to erasure
    Incorrect. The user wants to correct their data, not delete it.
  • Right to restriction
    Incorrect. Restriction means limiting how data is used. The user wants to fix incorrect data.
  • Right to rectification
    Correct. Article 16 gives users the right to have inaccurate personal data corrected without undue delay.
  • Right to object
    Incorrect. The right to object is about stopping certain types of processing, not correcting data.

Q4: Are GDPR user rights optional features that developers can choose to implement?

  • Yes, they are nice-to-have features
    Incorrect. GDPR rights are legal requirements. Failing to support them can lead to fines and legal action.
  • No, they are mandatory where applicable
    Correct. GDPR user rights are legally binding. Your system must support them where they apply.
  • Only the right to erasure is mandatory
    Incorrect. All 8 rights are mandatory where applicable, not just erasure.

Quiz: Implementing User Rights

Q1: A user wants to download their data as JSON so they can switch to a competing service. Which right applies?

  • Right of access
    Incorrect. Access (Article 15) lets users get a copy of their data. Portability goes further by requiring a structured, machine-readable format for reuse elsewhere.
  • Right to data portability
    Correct. Article 20 gives users the right to receive their data in a structured, commonly used and machine-readable format. This allows them to transfer it to another service.
  • Right to restriction
    Incorrect. Restriction (Article 18) limits how data is used. It does not relate to exporting data.
  • Right to erasure
    Incorrect. Erasure is about deleting data, not exporting it.

Q2: An algorithm automatically rejects a user's loan application. Under GDPR, what must you provide?

  • Nothing, automated decisions are always allowed
    Incorrect. Article 22 restricts fully automated decisions that significantly affect users. There are strict conditions on when this is allowed.
  • A way for the user to request human review
    Correct. Article 22 requires that you provide meaningful information about the logic involved and offer a way to request human intervention, express their point of view and contest the decision.
  • A full copy of the algorithm source code
    Incorrect. You do not need to share source code. You must provide meaningful information about the logic and allow human review.
  • An apology email
    Incorrect. An apology does not fulfill the requirements. Users have a right to human review and to contest the decision.

Q3: A user objects to receiving marketing emails. What must you do?

  • Continue sending for 30 more days
    Incorrect. There is no grace period. When a user objects to direct marketing, you must stop immediately.
  • Stop sending marketing emails immediately
    Correct. Article 21(2) states that when a user objects to processing for direct marketing purposes, processing must stop. This is an absolute right with no exceptions.
  • Ask them to confirm three times before stopping
    Incorrect. You cannot add hurdles to make it harder for users to exercise their rights. One objection is enough.
  • Ignore the request if they signed up voluntarily
    Incorrect. Even if the user originally consented, they have the right to object at any time. You must respect it.

Q4: A user disputes the accuracy of their data and asks you to pause processing until it is verified. Which right are they exercising?

  • Right to object
    Incorrect. Objecting (Article 21) is about stopping processing for specific reasons like marketing. This scenario is about pausing processing while data accuracy is being checked.
  • Right to restriction
    Correct. Article 18 gives users the right to restrict processing when they contest the accuracy of their data. You must pause processing until the data is verified.
  • Right to erasure
    Incorrect. The user is not asking for deletion. They want processing paused while accuracy is being verified.
  • Right to data portability
    Incorrect. Portability is about exporting data in a reusable format. It has nothing to do with pausing processing.
